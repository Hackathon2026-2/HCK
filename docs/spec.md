# YAWARAi 開発仕様書（テーマ「柔」）

> このファイルは Claude Code と Codex の両方が読む実装の正典。迷ったら §15 実装優先順位 から着手する。
> 最優先原則: 動くこと > 機能数 > 綺麗さ。完了 = Vercel のURLで実機（PC＋webカメラ）動作まで確認。

## 1. アプリ概要
**YAWARAi**（ヤワラ・アイ）。Webカメラ体験型ゲーム。怒ったAIが投げる「硬いもの（硬い言葉）」を体で避け、「柔らかいもの（優しい言葉）」を体で集めてAIの怒りを鎮める。名前は「柔」＋「AI」＋怒り・硬さが和らぐ意。

## 2. ターゲットユーザー
AIに「早くして」「直して」など強い言葉や無理な命令を投げがちな利用者。1分で「ぶつからず、柔らかく受け止めて鎮める」を体感させる。

## 3. テーマ「柔」との対応
- 体の柔: カメラ前で体を左右にしならせ、硬を避ける。
- 言葉の柔: 「ありがとう」「ゆっくりでいいよ」等を集めて鎮める。
- 関係性の柔: 命令の関係から思いやりの対話関係へ。沈静後のAIの感謝で表現。

## 4. コアループ / ゲームルール
- 画面上方からアイテムが落下。最下部の「キャッチ帯」に、体の左右位置で動くキャッチャーがいる。
- 柔がキャッチャーに重なる＝回収（怒り↓・スコア↑・コンボ＋）。
- 硬がキャッチャーに重なる＝被弾（怒り↑・スコア↓・コンボ0）。
- 怒りゲージが高いほど落下速度・出現間隔・硬の比率が上がる（＝難化）。柔を集めるほど鎮まり易化。
- 終了: `anger<=0` で沈静（クリア）、または制限時間到達でも終了 → 結果画面。

## 5. 画面遷移
`start` → `intro`（怒るAIの導入演出）→ `camera`（カメラ許可・操作説明）→ `playing` → `result`。
`camera` で許可が取れなければ keyboard/mouse 操作にフォールバックして `playing` へ。

## 6. UI構成（画面別）
- **start**: アプリ名、コンセプト一言、スタートボタン。
- **intro**: 怒ったAI（表情=angry）、なぜ怒っているかのセリフ、進むボタン。アニメ＋セリフでよい（実動画は不要）。
- **camera**: 自分のカメラ映像（ミラー表示）、「体を左右に動かして柔を集めよう」、ゲーム開始ボタン。
- **playing**: 背景に薄くカメラ映像＋ゲームcanvas（落下アイテム・キャッチャー）、HUD（スコア／怒りゲージ／残り時間／コンボ）、AIキャラ（表情はゲージ連動）。
- **result**: 最終スコア、しなやか度%、AIの状態（calm）、AIからの一言（生成）、「AIにもやさしくしよう」メッセージ、もう一度ボタン。

## 7. 使用する状態変数
毎フレーム更新するものは ref（再描画しない）、画面・HUD は React state に分ける。
```ts
type Phase = 'start'|'intro'|'camera'|'playing'|'result';
type AiMood = 'angry'|'irritated'|'neutral'|'calm';
type InputMode = 'camera'|'keyboard'|'mouse';

interface HudState { score:number; anger:number; shinayaka:number; timeLeft:number; combo:number; }
interface Item { id:number; kind:'soft'|'hard'; x:number; y:number; vy:number; label:string; }
interface GameRef { playerX:number; items:Item[]; lastSpawnMs:number; running:boolean; prevFrame?:ImageData; }
```
- React state: `phase`, `hud(score/anger/shinayaka/timeLeft/combo)`, `aiMood`, `aiMessage`, `inputMode`, `cameraDenied`
- ref: `playerX`(0..1), `items[]`, スポーン/タイマー管理, 直前フレーム画素（動き検出用）
- HUD は毎フレームではなく 5〜10回/秒に間引いて state 反映。

## 8. 当たり判定の考え方（＋動き検出）
動き検出（MVP。骨格認識は使わない）:
1. カメラ映像をミラーし、小さなオフスクリーンcanvas（例 160×120）へ毎フレーム描画。
2. 直前フレームとのグレースケール差分 → 閾値で2値化（静止背景は無視できる）。
3. 変化画素の横方向の重心を算出 → 0..1 に正規化 → `playerX`。EMA（α≈0.3）で平滑化。
4. ジッター時は画面を左/中/右の3レーンに離散化。カメラ不許可・低速時は keyboard ←→ / mouse X にフォールバック。

当たり判定:
- 最下部に高さのある「キャッチ帯」。キャッチャーは `playerX` 中心・幅≈120px の矩形。
- アイテムは毎フレーム落下。アイテムの y がキャッチ帯に入り、かつ `|item.x - playerXpx| < (catcherW+itemW)/2` で接触。
- soft接触→回収、hard接触→被弾。判定後そのアイテムは消す。
- 検出は縮小canvasで軽量に、描画はゲームcanvasでフルサイズ（requestAnimationFrame）。

## 9. 計算ルール（初期値・調整前提）
- 怒り `anger`: 0–100、開始 100、`clamp(0,100)`。`anger<=0` で沈静（クリア）。
- 残り時間 `timeLeft`: 90秒。0で終了 → 結果。
- 柔回収: `score += 100 * comboMult`; `anger -= 7`; `combo += 1`。
- 硬被弾: `score = max(0, score-50)`; `anger += 12`; `combo = 0`。
- コンボ倍率: `comboMult = 1 + Math.floor(combo/5)*0.5`。
- 落下速度(px/s): `fallSpeed = 180 + anger*1.2`。
- 出現間隔(ms): `spawnInterval = clamp(350, 900 - anger*4, 900)`。
- 硬出現率: `hardRatio = 0.35 + 0.4*(anger/100)`（怒り高→硬多）。
- 難易度はすべて `anger` から導出（別変数を持たない）。
- 表情閾値: `>70 angry / 40–70 irritated / 10–40 neutral / <10 calm`。
- しなやか度(%): `soft+hard>0 ? round(soft/(soft+hard)*100) : 100`（soft=回収数, hard=被弾数）。
- 公平性: 同一x・近接yに soft と hard を同時生成しない（避けられない配置を作らない）。

## 10. AI活用（審査基準「AI活用度」を稼ぐ）
- 動的セリフ（推奨・キー疎通できる場合）: LLM で「開始の怒り口上」と「結果の個別コメント（しなやか度・回収数・被弾から）」を生成。`app/api/ai/route.ts`（サーバー）でキーを秘匿して呼ぶ。**毎フレーム呼ばない**。開始時と結果時のみ非同期。
- 音声: ElevenLabs で上記セリフ＋固定の節目セリフ（怒り/エスカレート/沈静）。実行時TTS依存を避けるなら固定セリフは事前生成して `public/audio/` に置き、閾値で再生。結果コメントは画面テキストのみでも可。
- 音楽: Suno の BGM（怒り用・静用の2系統）を事前生成 → ファイル再生。
- フォールバック（キー間に合わず）: セリフは怒り段階別の定型文プールから選択。ElevenLabs音声＋Suno音楽で AI活用 は担保される。

## 11. 技術スタック / ディレクトリ / API
- Next.js(App Router)+TypeScript+Tailwind、Vercel、getUserMedia、Web Audio、Canvas。
- `app/page.tsx`（フェーズ管理）, `app/api/ai/route.ts`（口上/結果生成→任意でElevenLabs）, `components/`（各画面・AIキャラ・HUD・GameCanvas）, `lib/`（detection / collision / spawn / score / types）, `public/audio/`（Suno/ElevenLabs音源）。
- APIキーは `.env.local` ＋ Vercel 環境変数。`.gitignore` に `.env*`。コミット禁止。

## 12. 技術的リスクと回避策（要対応順）
1. 動き検出の不安定さ → 縮小canvas差分＋EMA平滑化＋開始1〜2秒キャリブレーション、3レーン/keyboard/mouse へ多段フォールバック。
2. APIキー未確定 → プロバイダ確定＋疎通を最優先。間に合わなければ定型文＋ElevenLabs。LLMは開始/結果のみ非同期。
3. 音の自動再生ブロック → 必ずスタートボタン操作で解錠（`AudioContext.resume()`）。
4. React再描画でカクつく → ループは canvas＋ref、React state は phase＋HUD（間引き更新）。
5. カメラ拒否・他端末 → 拒否時フォールバック、操作は「柔を集め硬を避ける」だけで説明不要に（審査員が自分で触っても成立）。
6. オンライン配信で音が届かない → 画面共有時「タブの音声を共有」。入力は体の動き＝マイク不要なので通話との競合なし。

## 13. MVP機能（必ず）
カメラ表示（ミラー）／`playerX` 左右判定／soft・hard出現／当たり判定／スコア加減・怒り増減／怒り連動のAIセリフ・難易度変化／結果画面（スコア・しなやか度・AI一言）。

## 14. 拡張機能（余裕があれば）
トゲトゲ言葉ボタン（押すと怒り↑・硬増＝関係が硬くなる表現）／AI表情アニメ／導入アニメ強化／コンボ演出／最後に優しい言葉を入力してAIに届ける／LLM動的セリフのリッチ化。

## 15. 実装優先順位（段階・各段でデプロイ確認）
1. フェーズ骨組み＋カメラ表示（ミラー）＋Vercel表示確認。
2. `playerX` 検出（まず keyboard ←→ でゲーム成立 → その後カメラ差分に差し替え）。
3. アイテム落下＋キャッチ帯の当たり判定（soft のみ → hard 追加）。
4. スコア・怒り・しなやか度・終了条件（怒り0/時間切れ）。
5. 怒り連動の AIセリフ（まず定型文）・難易度変化・HUD。
6. 結果画面＋AIの一言（定型 → 可能なら LLM）。
7. 音（スタートで解錠、閾値で再生）。
8. （拡張）トゲトゲ言葉ボタン・演出・動的セリフ。

## 18. 発表用説明文
「テーマは『柔』。私たちは柔を、硬いものに正面からぶつからず、しなやかに受け流す力と捉えました。普段、私たちはAIに無理なお願いや強い言葉を投げがちです。そこで、怒ってしまったAIが投げる『硬い言葉』を体で避け、『優しい言葉』を集めて鎮める Webカメラ体験ゲームを作りました。体のしなやかさ、言葉のやわらかさ、そしてAIとの関係のやわらかさを、遊びながら体験できます。」
