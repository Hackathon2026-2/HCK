# コーディング規約（Next.js + TypeScript）

- スタック確定: Next.js (App Router) + TypeScript + Tailwind CSS。ホスティングは Vercel。
- まず Server Component / `useState`。状態管理ライブラリは入れない。
  理由: 3時間ではライブラリの学習・設定コストが回収できない。
- 命名: コンポーネント PascalCase / 関数・変数 camelCase / ファイルは Next.js 慣習に従う。
- 早期 return とベタ書きを恐れない。共通化は同じものが2回出てから。

## 型（崖にはガードレール）
`any` は時間が無いとき限定で許容。ただし **API の出入口（route handler の入出力）には必ず型を置く**。
理由: 境界の型崩れは実行時に静かに壊れ、デモ中に露見する。

## LLM / 外部API
- サーバー処理（LLM呼び出し含む）は `app/api/.../route.ts` に置く。
- APIキーは `.env.local`（ローカル）と Vercel の環境変数（本番）から読む。
  **キーは絶対にコミットしない**。`.gitignore` に `.env*` が入っていることを必ず確認する。
  （1文字の漏れが事故になる箇所なので、ここは理由ではなく無条件で厳守）

## 完了の定義（契約）
「ビルドが通った」「ローカルで動いた」は完了ではない。完了とは:
1. 対象機能が **Vercel のデプロイURL** 上でブラウザから動く
2. デモシナリオの該当ステップを実際にクリックして通ることを確認した（スクショ or 手動）

スタブ実装・中身が空の関数・型だけ通って動かないものを「完了」と報告しない。

## やらないこと（時間泥棒の遮断）
- コア機能でない認証・ログイン
- 不要な DB / 複雑なスキーマ
- テストの網羅（デモ前にクリティカルパスを手で通せば十分）
- 過度なコンポーネント分割・凝った状態管理
- 完璧なレスポンシブ/アクセシビリティ（デモに映る範囲だけ整える）
