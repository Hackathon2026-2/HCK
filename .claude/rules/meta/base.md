# 基本ルール（全タスク共通・常時適用）

このリポジトリは全日本AIハッカソン2026 予選4thラウンド用。制作時間は約3時間、テーマは当日発表、その場で作ったものだけが評価される。すべての判断は「3時間で動くデモに着地するか」で行う。

## 優先順位（迷ったら上が勝つ）
動くデモ ＞ 機能の数 ＞ コードの綺麗さ。
デモシナリオに映らない作業（テスト網羅・完璧なエラー処理・リファクタ・将来拡張）は後回し。
理由: 評価されるのは画面で動く成果だけで、内部品質は加点されない。

## 過剰設計をしない
抽象化・汎用化・設定可能化・将来のための拡張ポイントは作らない。今必要な最小実装だけ書く。
理由: 抽象化は今のトークンと時間を消費し、3時間では回収できない。

## わからないことは質問する（推測で埋めない）
仕様の解釈・ライブラリの挙動・APIの形が不確かなら、コードを書く前に質問する。
未確認の数値・名前・前提を、事実のように使わない。
理由: 推測で進めて後で巻き戻すのが3時間で一番高くつく。

## 問題のある指示には反論する
盲目的に従うのは失敗。スコープを膨らませる指示・デモに不要な作り込み・時間を溶かす方針には、
代替案を添えて一言指摘する。

## レビュー・調査は中立の枠で頼む / 行う
「バグを探せ」ではなく「ロジックを追って報告しろ」。
「問題がある」前提で頼むと、無い問題まで作り出すので、中立に検証する。

## 作業領域の分担（衝突を物理的に消す）
- にゃんこ2 (Claude Code): ロジック/データ/API → `lib/`, `app/api/`
- 相方 (Codex): UI/デザイン/動画/磨き込み → `components/`, スタイル
相手の領域には断りなく手を入れない。共有ファイル（`app/layout.tsx`・共有の型・`package.json`・設定）を触るときは一言。

## Git
- `main` 一本のトランクベース。コミットは小さく頻繁に。push 前に必ず `git pull --rebase`。
- `main` は常にデプロイ可能に保つ。壊れる実験は短命ブランチで（Vercel がプレビューURLをくれる）。
- コミットメッセージは `<type>: <内容>` の1行（type = feat/fix/style/chore/docs）。

## 中間成果はファイルに出す
調査結果・設計メモ・長い出力は会話に貼らずファイルに書く。コマンド出力もサマリーだけ見て詳細はログへ。
理由: セッションが進むと前半の内容は圧縮されて消える。ファイルに逃がせば読み直せる。

## 完了報告の形式（曖昧な「やりました」は禁止）
作業のたびに次の3つを、数値を添えて書く。
- 変更したもの（例: 3ファイル / +47行、`page.tsx`・`route.ts`・`Button.tsx`）
- 触っていないもの
- 懸念点
